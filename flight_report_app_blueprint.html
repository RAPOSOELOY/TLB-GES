How it works
🔢
Server-side counter
An atomic increment in the database ensures each number is unique and sequential — no race conditions.
📴
Offline mode
When offline, a temporary local ID is used. The final sequential number is assigned on next sync.
🔒
Immutable once signed
Once a report is signed, its number is permanently locked. Deletion is blocked at the API level.
📋
Audit trail
Every gap in the sequence triggers an alert in the admin panel for compliance tracking.
External signatures & authentication
External personnel (mechanics, inspectors) can sign documents using a TOTP-based identity verification flow — no app installation needed on their device.
1
Pilot requests external signature
Selects the signer's role and taps "Request signature". The app generates a time-limited signing session.
2
Signer receives notification
A push notification or SMS is sent with a deep link and a 6-digit TOTP code valid for 30 seconds.
3
Identity verified
The signer enters their TOTP code on the iPad. The server validates it against the signer's registered secret.
4
Signature recorded
Name, role, timestamp (UTC), and device fingerprint are permanently attached to the document hash.

Key security constraints
Signed reports are cryptographically sealed — a SHA-256 hash of the full document is stored at signing time. Any modification breaks the hash.
Timestamps are server-side UTC — the device clock cannot be manipulated to alter the recorded signature time.
Sequential numbering uses a PostgreSQL sequence — atomic, gap-free, resistant to concurrent writes.
TOTP codes expire in 30 seconds — even if intercepted, they cannot be reused or replayed.
View-only mode for signed reports is enforced at both the UI layer (all fields disabled) and the API layer (PUT/PATCH blocked).
